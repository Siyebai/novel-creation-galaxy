## 2. 踩坑全记录（10大坑）

### 🕳️ 坑1：PowerShell文本替换的换行符陷阱

**现象：** PowerShell的 -replace 操作符 + [regex]::Escape() 在处理从文件中读取的文本时，换行符\r\n与\n不匹配，导致大部分替换没有生效。

**根因：** 
- 文件中实际使用的是 \r\n（Windows换行）
- PowerShell -replace 期望 \n
- [regex]::Escape() 不会自动处理换行符差异

**解决方案：** 
- 改用Node.js的 eplace() 方法直接操作字符串
- 用 indexOf() 先检测匹配位置再替换
- 把文本写入文件→JS读取→替换→写回（避免PowerShell的字符串解析问题）

**内化规则：** ⚡ 涉及中文字符串替换时，优先用Node.js文件脚本，避免PowerShell字符串处理。

### 🕳️ 坑2：PowerShell字符串中的引号冲突

**现象：** 在PowerShell双引号字符串 "..." 中嵌套中文双引号 " 时，PowerShell会误解释为字符串结束。

**根因：** PowerShell用 " 作为字符串分隔符，内部的双引号需要用 "" 转义，但包含中文文本时容易遗漏。

**解决方案：**
- 用单引号字符串 '...' 避免解析
- 用单引号here-string @'...'@ 写入JS文件（不解析任何内容）
- 把中文双引号 " 替换为中文引号 「」

**内化规则：** ⚡ PowerShell脚本中，中文字符串优先用单引号；需要写入文件时用 @'...'@ heredoc。

### 🕳️ 坑3：Node.js代码在PowerShell中的 $$ 展开

**现象：** 在PowerShell中写内联Node.js代码 
ode -e "..." 时，page.eval 中的 $$ 被PowerShell解释为自动变量，导致 page.eval 而非 page.eval。

**根因：** PowerShell中 $$ 表示"当前会话中最后一个命令的最后一个token"。

**解决方案：** 
- 用 page.evaluate() 替代 page.eval()（执行自定义JS）
- 把代码写入 .js 文件执行，避免PowerShell解析

**内化规则：** ⚡ 所有Node.js/Python脚本必须写入文件后执行，绝不使用内联 -e 方式。

### 🕳️ 坑4：GitHub Secret Scanning拦截Token上传

**现象：** 上传 GITHUB_CONFIG.md 到公开仓库时被拒绝（409），原因是文件内容包含GitHub token。

**根因：** GitHub的Secret Scanning功能会自动扫描公开仓库中的敏感信息，发现token会阻止上传。

**解决方案：** 
- 将token替换为占位符 [已加密存储]
- token仅存储在本地安全位置
- 创建单独的脱敏版本上传

**内化规则：** ⚡ 永远不要将任何token/密码上传到公开仓库。用脱敏版本+本地存储。

### 🕳️ 坑5：git clone超时（网络限制）

**现象：** git clone https://github.com/... 超时（Connection timeout），无法克隆仓库。

**根因：** 中国大陆访问GitHub的网络限制，HTTPS连接慢或不稳定。

**解决方案：**
- 改用GitHub API直接上传文件（PUT /repos/{owner}/{repo}/contents/{path}）
- API访问正常（HTTPS端口443，但git协议不同）

**内化规则：** ⚡ 快速操作优先用GitHub API（node.js https），需要完整git历史才用git clone。

### 🕳️ 坑6：npm全局包不能通过require直接加载

**现象：** equire('@anthropic-ai/claude-code/package.json') 失败，虽然全局已安装。

**根因：** npm全局包的安装路径（%APPDATA%/npm/node_modules）不在 equire() 的搜索路径中，需要用 
pm root -g 获取路径。

**解决方案：**
- 用 
px 命令执行（自动查找全局包）
- 或在项目目录 
pm link <package> 链接到本地

**内化规则：** ⚡ 全局工具用 
px <tool> 调用；本地项目工具用 
pm install 安装到项目目录。

### 🕳️ 坑7：Playwright无头浏览器触发知乎安全验证

**现象：** 使用Playwright访问知乎时，出现「安全验证」页面要求人机验证，即使Cookie有效。

**根因：** 
- 知乎检测到无头浏览器的指纹（webdriver、userAgent等）
- 异常IP触发风控
- 自动化操作被识别

**解决方案：**
- 使用 headless: false 有头模式
- 设置真实的UserAgent
- 添加更多浏览器指纹伪装（--disable-blink-features=AutomationControlled）
- 最终方案：让用户在真实Edge浏览器中操作

**内化规则：** ⚡ 中国主流网站（知乎/微博/淘宝等）有强反爬机制，自动化方案需要预留验证码处理路径。

### 🕳️ 坑8：浏览器验证码无法自动通过

**现象：** 等待120秒后验证未完成，脚本超时。

**根因：** 
- 弹出的浏览器窗口可能被遮挡（用户未看到）
- 滑块/点选验证需要人工干预，无法自动完成

**解决方案：**
- 改为混合模式：脚本准备所有内容，用户手动在已有浏览器中发布
- 用截图展示当前状态给用户

**内化规则：** ⚡ 涉及登录验证的自动化任务，始终准备「纯自动→半自动→手动」三级备用方案。

### 🕳️ 坑9：知乎API登录返回403

**现象：** 直接调用 POST /api/v3/oauth/sign_in 返回403 "请求参数异常"。

**根因：** 
- 知乎API需要计算签名（signature），算法未知
- 缺少必要的 x-api-version 等header
- 可能需要先获取 _xsrf token

**解决方案：** 改用浏览器自动化方案（但遇到坑7和坑8）

**内化规则：** ⚡ 国内大厂的API签名算法复杂，优先用浏览器自动化而非API直接调用。

### 🕳️ 坑10：Edge用户数据目录被占用

**现象：** Playwright的 launchPersistentContext 使用Edge用户数据目录时报错，无法获取锁。

**根因：** Edge已经在运行中，用户数据目录被进程锁定，无法被第二个实例同时使用。

**解决方案：**
- 使用新的临时profile目录
- 或用CDP连接到已在运行的Edge（需要预先开启debug端口）

**内化规则：** ⚡ 浏览器用户数据目录同一时间只能被一个进程使用。如有已运行浏览器，用新profile或CDP连接。

## 5. 行为模式优化

### 5.1 新增效率规则

`
规则1️⃣  中文文本替换 → 优先用Node.js文件脚本（避免PowerShell字符串问题）
规则2️⃣  批量文件上传 → 用GitHub API（避免git clone网络限制）
规则3️⃣  知乎/web自动化 → 准备三级方案（纯自动→半自动→手动）
规则4️⃣  Token/密码 → 永远不上传公开仓库，用脱敏版替代
规则5️⃣  Node.js脚本 → 写文件执行，不用内联 -e 模式
规则6️⃣  npm工具调用 → 统一用 npx（避免require路径问题）

### 5.2 任务优先级策略

`
接到任务 → 快速评估 → 选择方案：

  ├─ 需要验证码 → 混合方案（脚本准备+用户处理验证）
  └─ 需要登录态 → 优先复用已登录浏览器Cookie
